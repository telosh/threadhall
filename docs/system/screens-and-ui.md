## 代表的ユーザーフロー（文章）

1. **主催**: サインイン（UI-02）→ 組織作成／選択（UI-03）→ イベント作成 `draft`（UI-06/07）→ サテライト計画通りプロビジョニング（UI-08）→ `live` 公開 → トークン配布（UI-09）→ 会場でログ収集（UI-11/12）→ 終了後 `archived`（UI-07）→ 公開閲覧（UI-13）。
2. **メンバー**: 日常は長命スレッド（UI-05）。イベント中はトークンまたはロールに応じ UI-11 に到達。
3. **トークンのみ参加者**: QR／リンクからクレーム取得（UI-09 経由で発行された URL）→ UI-11 で最低限の書込。**ログイン必須にしない構成も許容**（セキュリティはサーバ検証で担保）。

## UI で必ず示すべきシステム上の事実（最重要）

| トピック | ユーザーに見せる／言語化する内容 |
|----------|----------------------------------|
| 書込の正 | 「このイベントへの記録は**参加トークン**が必要です」— Geo は補助であることを補足で |
| アーカイブ | 「締切後は**新規投稿・修正・削除はできません**（閲覧のみ）」 |
| トークン | 有効期限・失効条件・第三者共有の注意 |
| データ分離 | 主催向けヘルプで「イベント記録は専用 DB に分離」と短く説明可能 |

## 権限とのクイック対応

- [`capabilities-by-role.md`](capabilities-by-role.md) の表と、上表の「主なペルソナ」を 1:1 で突き合わせ、**メニュー表示条件**を決める。
- **未ログイン**で触れてよいのは UI-01, UI-13（公開時）, UI-02, UI-14 が基本。それ以外はサインインまたはトークン必須。

## 技術メモ（フロント配置）

静的ホストが Firebase / GCS+LB のどちらでも、**認証コールバック URL** と **環境別 Secret** は [`gcp-architecture.md`](gcp-architecture.md) に合わせて管理する。
